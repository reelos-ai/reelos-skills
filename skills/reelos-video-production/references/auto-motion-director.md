# 自动动效导演

## 目标

自动动效导演不是增加动画数量，而是让每个 TTS beat 获得最能解释当前关系、且最稳定的视觉动作。适用于产品宣传、科普、系统模型、商业讲解、SketchTalk 和 `slides-video`。

固定流程：

```text
真实 TTS cue → 语义分类 → 生成候选 → 硬约束过滤 → 100 分评分 → 去重复 → 写入 Beat Motion Map → 关键帧返修
```

## 一、语义分类

每个 beat 只设置一个主语义；确有复合关系时，再设置一个次语义。

| 语义类型 | 识别问题 | 首选视觉语法 |
| --- | --- | --- |
| `trend` 趋势 | 是否描述随时间增减、转折或成本变化 | 曲线生长、柱状变化、数字递变、面积压缩 |
| `comparison` 对比 | 是否比较前后、两类模式或优劣 | 分屏、基线对齐、差异高亮、状态置换 |
| `causal` 因果 | 是否表达 A 推动、抑制或导致 B | 节点连线、路径传导、变量激活、反馈回路 |
| `process` 流程 | 是否存在明确先后顺序 | 路径绘制、步骤点亮、进度推进、阶段解锁 |
| `selection` 筛选 | 是否从多个候选中判断、过滤或收敛 | 卡片散开、淘汰变灰、聚焦框、候选收束 |
| `input` 输入 | 是否真实发生搜索、输入、命令或网址访问 | 打字、光标、回车、加载反馈 |
| `state` 状态 | 是否从一个稳定状态转到另一个状态 | 状态切换、颜色迁移、形态变换、仪表锁定 |
| `evidence` 证据 | 是否需要展示产品、数据、来源或实际结果 | 网站聚焦、截图扫描、数据标注、局部放大 |
| `conclusion` 结论 | 是否是核心判断、口号或 CTA | 减速、收束、锁定、留白、品牌定格 |
| `atmosphere` 氛围 | 是否只需要情绪和呼吸，不承担关系解释 | 极轻背景层、微弱网格、静态主题图层 |

先判断“观众需要看懂什么关系”，再选择效果。不要从已有特效列表反推内容。

## 二、候选生成

每个 beat 生成 2-3 个候选，其中至少一个是低复杂度方案。候选必须写清：

```ts
type MotionCandidate = {
  id: string;
  semanticType: 'trend' | 'comparison' | 'causal' | 'process' | 'selection' | 'input' | 'state' | 'evidence' | 'conclusion' | 'atmosphere';
  primaryAction: string;
  supportAction?: string;
  transition: string;
  complexity: 1 | 2 | 3;
  evidenceSource?: string;
};
```

候选示例：

- “构建成本逼近于零”：下降曲线、数字递减、柱形压缩。
- “哪些是信号，哪些是噪音”：卡片筛选、信噪比分界、候选收束。
- “事件如何推动资产”：因果路径、节点脉冲、变量传导。
- “打开 macro.reelos.ai”：逐段打字、浏览器地址栏输入、品牌 CTA 定格。

## 三、硬约束过滤

候选触发以下任一条件时直接淘汰，不进入评分：

- 图表没有真实数据、可验证方向或明确“示意”标记。
- 口播没有输入行为，却使用打字机效果。
- 口播没有趋势关系，却使用增长曲线。
- 口播没有因果或流程，却使用路径传导。
- 同一 beat 同时安排两个以上主运动对象。
- 大号中文主标题持续缩放、抖动、模糊或小数像素漂移。
- 高密度图表同时进行大幅镜头移动。
- 第一帧依赖淡入才出现标题或品牌。
- 使用 CSS animation、浏览器计时器或不可按帧复现的随机数。
- 效果需要牺牲文字可读性、真实产品证据或渲染稳定性。

## 四、100 分评分

对通过硬约束的候选评分：

| 维度 | 权重 | 判断标准 |
| --- | ---: | --- |
| 语义准确 | 35 | 动作是否直接表达当前 beat 的关系，而非泛化装饰 |
| 理解效率 | 25 | 是否比纯文字更快让观众理解，是否减少认知负担 |
| 节奏匹配 | 15 | 是否能在 cue 时长内完成，并与停顿、重音和转折一致 |
| 品牌一致 | 15 | 是否符合当前模板、色板、字体和运动骨架 |
| 技术稳定 | 10 | 是否可按帧复现、渲染成本合理、不会抖动或重排 |

计算：

```text
score = semantic + clarity + rhythm + brand + stability
```

- 只采用总分不低于 75 的候选。
- 最高分候选成为 `selected`。
- 分差小于 3 分时，选择复杂度更低者。
- 所有候选都低于 75 时，回退到稳定排版、局部高亮或静态证据，不强行动画。

## 五、连续性与去重复

单个候选得分高，不代表放进全片后仍然最好。选择完成后再做全局检查：

- 同一种 `primaryAction` 不得连续超过两个 beat。
- 相邻 Slide Story 页面不得连续使用同一构图和同一主动作。
- 宣传短片每 2-4 秒出现一次有意义的视觉状态变化。
- 教学/科普每 3-6 秒变化一次；长视频低密度段可放宽到 4-8 秒。
- 一个 scene 内最多一个主动作和两个低强度辅助动作。
- 图表、截图和大标题轮换承担视觉重心，避免全片只有卡片或大字。
- 转场表达上下文：延续用平移，深入机制用推近，因果用沿路径推进，筛选用收束，结论用减速锁定。

如最高分方案导致连续重复，对其施加 8-15 分重复惩罚，再从候选中重新选择。

## 六、MotionDecisionPlan 契约

在实现组件前，生成可审计的决策表：

```ts
type MotionDecisionPlan = {
  sceneId: string;
  cue: string;
  startFrame: number;
  endFrame: number;
  semanticType: string;
  candidates: Array<MotionCandidate & {
    semantic: number;
    clarity: number;
    rhythm: number;
    brand: number;
    stability: number;
    total: number;
    rejectedReason?: string;
  }>;
  selected: string;
  fallback: string;
  focus: 'caption' | 'chart' | 'path' | 'node' | 'product' | 'brand';
  transition: string;
};
```

`MotionDecisionPlan` 可以写入项目的 `motion-plan.ts`、JSON 或制作记录。Beat Motion Map 读取 `selected`，不要在组件里重新随机挑选效果。

## 七、自动审片与返修

至少导出：首帧、每个 scene 边界、每幕 15%/50%/85% 和结尾帧。按以下顺序检查：

1. **语义覆盖**：静音看画面，能否识别趋势、因果、流程或结论。
2. **重复度**：连续两个以上 beat 是否仍然只是相同卡片飞入或大字替换。
3. **视觉负荷**：主字幕、图表、截图和装饰是否争抢注意力。
4. **时序**：动作是否落在真实 cue 附近，是否早于口播泄露答案。
5. **稳定性**：首帧是否完整，是否存在空白、抖动、遮挡、溢出和黑帧。

自动返修规则：

| 问题 | 优先修正 |
| --- | --- |
| 语义不清 | 换成关系更直接的图表、路径或状态变化 |
| 连续重复 | 改变视觉重心，不只是换 easing |
| 信息拥挤 | 删除辅助动作，降低文字量，拆分 beat |
| 动效太弱 | 增加状态差异，不增加抖动或粒子 |
| 动效太强 | 主体稳定，变化下放到路径、节点或局部焦点 |
| cue 偏差 | 调整 cue offset 或 timing，不靠估算时长 |
| 渲染过慢 | 预烘焙背景，减少实时 blur、滤镜和超大透明 PNG |

## 八、验收门槛

- 所有核心 beat 都有明确语义类型。
- 需要视觉解释的 beat，合格动效覆盖率不低于 80%。
- 选中候选总分不低于 75。
- 同一种主效果连续出现不超过两次。
- 第一帧完整，结论和 CTA 有稳定停留。
- 关键图表和路径在 15%/50%/85% 三帧能看出状态差异。
- 文字始终可读，没有因动效发生布局跳动。
- 所有 cue 来自真实 TTS timing；用户改稿后重新生成 TTS、timing 和 MotionDecisionPlan。

最终标准：动画不是让画面更忙，而是让口播中看不见的关系被直接看见。
